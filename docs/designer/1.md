# Technical Design Document - Iteration 1

## Overview

This document outlines the architecture improvements for the TUI Chatbot based on patterns from `refs/pi-mono/` (packages/ai and packages/agent).

---

## Key Patterns from pi-mono

### 1. EventStream Architecture

**Current State:**
- Uses asyncio.Queue with polling loop (0.1s timeout)
- Simple push/end/error interface

**pi-mono Pattern (Preferred):**
- Pull-based async iteration using waiters/resolvers
- `isComplete` callback to detect terminal events
- `extractResult` callback to compute final result from terminal event
- No polling - purely event-driven

**Benefits:**
- More efficient (no polling timeouts)
- Cleaner backpressure handling
- Standardized terminal event detection

### 2. Event Protocol (AssistantMessageEvent)

**Current State:**
- Simple EventType enum: REASONING_TOKEN, CONTENT_TOKEN, STATS, DONE, ERROR
- Events carry raw data only

**pi-mono Pattern (Preferred):**
```
start -> text_start -> text_delta* -> text_end
      -> thinking_start -> thinking_delta* -> thinking_end
      -> toolcall_start -> toolcall_delta* -> toolcall_end
      -> done | error
```

Each event carries:
- `type`: Event type
- `partial`: Current AssistantMessage snapshot
- `contentIndex`: Index of content block being updated
- `delta`: Incremental change (for delta events)
- `content`/`toolCall`: Final value (for end events)

**Benefits:**
- Fine-grained streaming updates
- Content block indexing for multi-block responses
- Structured final results in terminal events

### 3. Message Types

**Current State:**
- Simple dict-based messages: `{"role": "user", "content": "..."}`
- No structured content blocks

**pi-mono Pattern (Preferred):**
```python
class TextContent:
    type: "text"
    text: str

class ThinkingContent:
    type: "thinking"
    thinking: str
    thinkingSignature: Optional[str]
    redacted: bool

class ToolCall:
    type: "toolCall"
    id: str
    name: str
    arguments: dict

class UserMessage:
    role: "user"
    content: str | list[TextContent | ImageContent]
    timestamp: int

class AssistantMessage:
    role: "assistant"
    content: list[TextContent | ThinkingContent | ToolCall]
    api: str
    provider: str
    model: str
    responseId: Optional[str]
    usage: Usage
    stopReason: StopReason
    errorMessage: Optional[str]
    timestamp: int
```

**Benefits:**
- Type-safe content handling
- Standardized metadata (usage, costs, timestamps)
- Support for multi-modal content

### 4. Provider Registry Pattern

**Current State:**
- Hardcoded OpenAI client in Daemon class

**pi-mono Pattern (Preferred):**
- Registry-based provider system
- Lazy loading via factory functions
- Consistent `stream()` and `streamSimple()` interface
- Model-based configuration (provider, baseUrl, api, compat settings)

**Benefits:**
- Pluggable provider architecture
- Easy to add new providers
- Clean separation of provider-specific logic

### 5. Agent Loop Design

**Current State:**
- No agent loop - direct streaming from Daemon

**pi-mono Pattern (Preferred):**
- Event-driven agent loop with lifecycle events:
  - `agent_start` / `agent_end`
  - `turn_start` / `turn_end`
  - `message_start` / `message_update` / `message_end`
  - `tool_execution_start` / `tool_execution_update` / `tool_execution_end`
- Configurable tool execution: sequential vs parallel
- Steering messages and follow-up messages support
- beforeToolCall / afterToolCall hooks

**Benefits:**
- Structured conversation management
- Extensible tool system
- Support for complex agent workflows

## Architecture Recommendations

### Module Structure
```
src/tui_chatbot/
├── __init__.py
├── main.py              # Entry point, CLI parsing
├── types.py             # Core message and content types
├── events.py            # Event protocol definitions
├── event_stream.py      # EventStream implementations
├── models.py            # Model configuration and registry
├── providers/
│   ├── __init__.py
│   ├── registry.py      # Provider registration
│   ├── base.py          # Provider interfaces
│   └── openai.py        # OpenAI provider implementation
├── agent/
│   ├── __init__.py
│   ├── types.py         # Agent-specific types
│   ├── loop.py          # Agent loop implementation
│   └── tools.py         # Built-in tools
├── daemon.py            # High-level API interface
├── frontend.py          # TUI rendering
└── shell.py             # Command routing and REPL
```

### Key Design Decisions

1. **Type Safety:** Use dataclasses with proper type hints throughout
2. **Event-Driven:** All async operations emit structured events
3. **Abort Signal:** Pass AbortSignal through all async operations
4. **Provider Agnostic:** Clean abstraction over different LLM providers
5. **Extensible:** Easy to add new providers, tools, and message types
6. **Testable:** Mock-friendly interfaces, no hidden dependencies

### Migration Path

1. Create new modules alongside existing code
2. Port main.py functionality incrementally
3. Maintain backward compatibility during transition
4. Remove old code once new implementation is stable
