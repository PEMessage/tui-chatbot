**设计方案:**

```python

    context: List[AgentMessage],
    config: AgentLoopConfig,
    signal: Optional[AbortSignal],
    emit: AgentEventSink,
) -> AssistantMessage:
    """
    流式获取助手响应 - 集成 Provider.
    
    流程:
    1. 从 config 获取 Provider (或从 ProviderRegistry)
    2. 转换消息格式为 Provider 接受的格式
    3. 调用 Provider.stream_chat() 获取事件流

    5. 返回完整的 AssistantMessage
```

   ```python
       from ..provider.registry import ProviderRegistry
       provider = ProviderRegistry.get("openai")
       if not provider:
   ```

   ```python
   def _convert_messages_to_provider_format(messages: List[AgentMessage]) -> List[dict]:
   ```

## 7. 设计决策记录

### 7.1 为什么使用向后兼容导出修复 H2

- **方案对比:**
  - A: 修改测试文件导入 → 需要修改多个测试文件
  - B: 在 main.py 添加兼容导出 → 单点修改，保持向后兼容

- **决策:** 选择方案 B，因为:
  1. 减少测试文件修改，降低回归风险
  2. 保持向后兼容，外部代码可以继续使用旧导入
  3. 符合开闭原则

### 7.2 为什么优先实现 Anthropic 而非 Ollama

- **原因:**
  1. Anthropic API 与 OpenAI 更相似，验证多 Provider 架构设计
  2. Claude 3.5/3.7 是主流模型，实际使用需求高
  3. Ollama 本地部署需要额外依赖 (ollama 包)，可作为 N=3 任务

### 7.3 Agent Loop 与 Provider 集成方式

- **决策:** AgentLoopConfig 中添加可选 provider 字段
- **理由:**
  1. 允许每个 Agent Loop 实例使用不同 Provider
  2. 支持动态 Provider 选择 (如根据模型自动选择)
  3. 回退到全局 ProviderRegistry 保持简单使用场景
